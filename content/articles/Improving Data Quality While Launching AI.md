Purpose of This Document

This document provides a clear, balanced view on how organisations should approach AI initiatives when data quality is imperfect. It addresses a common misconception—that all data must be fully remediated before AI can begin—and outlines the recommended “parallel track” approach supported across the leading data and AI strategy frameworks.

1. AI Value Creation and Data Foundation Build Should Run in Parallel

Successful data-driven organisations do not wait for perfect data quality before starting AI work. Instead:

They improve the data foundation in targeted areas that enable high-value use cases.

They start AI initiatives early to surface data issues that matter most.

They avoid boiling the ocean with enterprise-wide remediation that has no immediate business value.

This parallel-track approach reflects the dual mandate found in modern data strategies:
defensive capabilities (quality, governance, security) and offensive capabilities (value creation, monetisation).

2. Different AI Use Cases Require Different Levels of Data Quality (QCS Framework)

Not all AI use cases demand the same level of data quality.
Using the QCS dimensions (Quality, Compliance, Speed):

Low-intensity use cases (e.g., simple BI dashboards, descriptive models) can begin with moderate-quality data.

Medium-intensity use cases (e.g., segmentation, forecasting) require improved and consistent quality.

High-intensity use cases (e.g., real-time models, GenAI used in customer-facing workflows) require robust quality and governance.

This means organisations can begin with lower-QCS requirements while building capabilities toward higher-QCS needs.

3. Early AI Projects Help Justify and Direct Data Quality Investments

Practical AI pilots often illuminate:

Data gaps

Inconsistencies

Ambiguous definitions

Lack of lineage or metadata

Broken ingestion or transformation logic

These insights are far more actionable than generic enterprise-wide "fix the data" programmes.

Early AI work therefore:

Creates visible business impact

Provides a concrete justification for data quality funding

Directs remediation toward the highest-value datasets

Helps the central data team secure long-term support

4. Data Quality by Design Still Applies — But It Is Iterative, Not Sequential

The principle of Data Quality by Design means:

quality must be built into processes early,

defects must be prevented rather than corrected later.

But “by design” does not mean “all quality issues must be solved first.”

Instead, organisations should:

Integrate quality practices into each new AI or analytics initiative.

Formalise quality expectations using Data Contracts.

Use each new initiative to mature governance and lineage.

Progressively uplift quality as the AI roadmap expands.

This is an iterative model, not a waterfall.

5. The DOM, EAV/TAV, and Product Thinking Reinforce the Parallel Approach

Three key frameworks support this viewpoint:

Data Operating Model (DOM)

The DOM encourages the creation of reusable data assets and requires data quality, documentation, and governance to be embedded into the asset creation process.
AI use cases consume these assets and feed back new quality requirements.

EAV/TAV Metrics

AI projects drive Expected Addressable Value (EAV), demonstrating why targeted quality improvements matter.
The gap between EAV and TAV helps prioritise which data issues to fix first.

Product Thinking

Treating data assets as products ensures they evolve over time:

versioned

improved

governed

monitored

—not “perfect before use.”

6. Practical Guidance for Executives and Data Leaders

Organisations should:

Identify high-value AI use cases with clear business sponsorship.

Assess the required QCS level for each use case.

Improve data quality only where necessary to enable these use cases.

Launch AI pilots early to validate value and expose data gaps.

Integrate quality, lineage, and governance into each new data asset, rather than treating them as separate projects.

Communicate the value created using EAV/TAV metrics to secure ongoing investment.

This ensures the data foundation improves because of AI initiatives—not in isolation from them.

7. Summary

AI cannot compensate for fundamentally poor data.

But waiting for perfect data quality before beginning AI is neither practical nor strategic.

The recommended approach is parallel progression:

uplift data quality where it matters most,

while launching AI pilots that validate value and reveal targeted quality gaps.

Frameworks such as QCS, DOM, EAV/TAV, and Product Thinking all support this progressive, iterative approach.

This balanced model enables organisations to deliver early AI value while building a robust, long-term data foundation.
