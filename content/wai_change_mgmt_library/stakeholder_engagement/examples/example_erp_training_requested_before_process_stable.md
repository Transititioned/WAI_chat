# ERP example: training requested before process is stable

## Situation

Business users keep asking for training on a process affected by the ERP upgrade.

The training request sounds reasonable, but the project team has not yet confirmed:

- which custom fields remain
- which status transitions have changed
- which exceptions still need workarounds
- whether the SOP reflects actual process behaviour
- who supports users if the transaction fails

## Pattern

This is a training request hiding process uncertainty.

The business may be asking for training because they want certainty. Training may not solve the issue until the process is stable enough to teach.

## Practical response

Separate the user need from the project uncertainty.

| User concern | Likely underlying gap | Better action |
|---|---|---|
| “We need training now” | Anxiety / uncertainty | Provide knowns/unknowns and training timeline |
| “What do we do when this fails?” | Support gap | Define escalation path and hypercare |
| “Which field do we use?” | System/process gap | Validate configuration and SOP |
| “Who owns the handoff?” | Role gap | Clarify responsibilities |

## Useful workplace language

“I agree training will be needed. Before we train the detail, we need to lock the process enough that we are not teaching something that changes next week.”

“Some of what is being called training is actually process and support clarity. Let’s separate those so we solve the right problem.”

## Good outcome

The project provides an interim knowns/unknowns update, completes SOP validation, confirms the support path, and then schedules practical training once the process is stable enough to teach.
