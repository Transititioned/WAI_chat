# ERP example: reports must/should/nice for go-live

## Situation

As go-live approaches, stakeholders request a large number of reports.

The requests include:

- compliance reports
- operational reports needed for day-one processing
- management dashboards
- convenience reports
- reports that duplicate old spreadsheet habits
- nice-to-have analytics ideas

The reporting list is too large to complete before go-live.

## Pattern

This is a must / should / nice finish issue.

The project needs to protect go-live without dismissing valid business needs.

## Practical response

Classify reports by go-live necessity.

| Report type | Category | Treatment |
|---|---|---|
| Compliance / statutory report | Must | Required for go-live or explicit risk acceptance |
| Operational report needed for daily processing | Must | Required for go-live |
| Management report needed for early post-go-live monitoring | Should | Prioritise if capacity allows or deliver soon after go-live |
| Convenience report replacing spreadsheet habit | Nice | Add to backlog |
| Analytics / optimisation idea | Nice | Add to improvement backlog |

## Useful workplace language

“I do not want to lose these reporting needs, but we need to separate what is required to operate safely on day one from what belongs in the post-go-live reporting backlog.”

“If this report is needed for compliance or daily operations, it may be a must. If it helps management insight but does not block operations, it may be a should.”

## Good outcome

The project completes the must-have reports, agrees risk treatment for any unresolved critical reports, and creates a visible post-go-live reporting backlog for the rest.
